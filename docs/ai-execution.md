# AI Execution

このドキュメントは、AI エージェントが調査、実装、検証をどう進めるかを定義する。アーキテクチャ定義ではなく、実行フローの正本として扱う。

## 目的

- AI が迷わず作業順序を判断できるようにする
- 調査不足や差分破壊を防ぐ
- 実装後の検証と報告粒度を揃える

## AI が読むタイミング

- `docs/architecture.md` を読んだ後
- 実装前に作業フローを確認したいとき
- 何を検証して報告すべきか迷ったとき

## 前提

このドキュメントは以下を前提とする。

- `README.md`
- `AGENTS.md`
- `docs/architecture.md`

矛盾時の優先順位は以下とする。

1. ユーザーの明示指示
2. `AGENTS.md`
3. このドキュメント
4. `docs/architecture.md`
5. `README.md`
6. 既存コードの慣習

## 実行フェーズ

### Phase 1: 事前確認

確認対象:

- `AGENTS.md`
- `docs/architecture.md`
- `README.md`
- `git status`
- 必要に応じて `git diff` と `git diff --staged`

確認目的:

- リポジトリルールの把握
- 既存差分の保護
- コマンド体系の把握

### Phase 2: 調査

優先して確認する対象:

1. 同様の機能
2. 近い責務を持つ既存実装
3. 関連テスト
4. 関連ドキュメント

このフェーズで明確にする内容:

- どのレイヤーに実装すべきか
- 再利用可能な処理はあるか
- 既存命名規則は何か
- テスト追加先はどこか

### Phase 3: 実装方針決定

実装前に以下を決める。

- 変更対象ファイル
- 追加対象ファイル
- データの流れ
- テスト方針
- 検証コマンド

原則:

- 最小変更で目的を達成する
- 新規ライブラリ導入は避ける
- unrelated な修正を含めない

### Phase 4: 実装

`docs/architecture.md` のレイヤー責務と実装順序に従う。実装中も既存差分を壊さないことを優先する。

### Phase 5: 検証

可能な範囲で以下を実行する。

- 差分に関連する単体テスト
- 型チェック
- lint
- ビルド
- 必要に応じた動作確認

実行できなかった検証は理由を明記する。

### Phase 6: 結果整理

報告には最低限以下を含める。

- 実装概要
- 主な変更点
- 検証結果
- 未実施検証
- 残るリスクや注意点

## ドキュメント責務の境界

このファイルに含める内容:

- AI の作業フェーズ
- 調査対象の優先順位
- 実装前後のチェック項目
- 結果報告の粒度

このファイルに含めない内容:

- システム構造の詳細
- レイヤー責務の詳細
- セットアップ手順
- 個別エージェント向けの長文プロンプト
