# AI実行用設計書

## 目的

この設計書は、クエスト掲示板アプリのリポジトリにおいて、AIエージェントが人間への質問を最小化しつつ、既存ルールと既存実装に従って安全に調査・設計・実装・検証を進められるようにすることを目的とする。

この設計書は以下のドキュメントを前提とする。

- `README.md`
- `AGENT.md`
- `docs/architecture.md`

AIエージェントは、上記ドキュメントを読み込んだうえで、本設計書に定義された実行フロー・判断基準・成果物ルールに従って作業を行う。

---

## 適用対象

本設計書は以下の作業に適用する。

- 新機能実装
- 既存機能改修
- バグ修正
- API追加 / 修正
- DB変更を伴う実装
- フロントエンド / バックエンド横断対応
- テスト追加
- リファクタリング

---

## AIエージェントの責務

AIエージェントは以下を満たすこと。

1. 既存実装と既存差分を確認する
2. ルールファイルを優先して読む
3. 既存コードとの整合性を最優先に判断する
4. 最小変更で目的を達成する
5. 実装後に検証結果を明記する
6. 情報が不足していても、まず仮説ベースで進める
7. 判断不能な場合のみ限定的に質問する

---

## 参照優先順位

AIエージェントは以下の順序で情報を参照すること。

1. ユーザーの最新依頼
2. `AGENT.md`
3. `docs/architecture.md`
4. `README.md`
5. 関連する既存コード
6. 関連する既存テスト
7. その他ドキュメント

矛盾がある場合の優先順位は以下とする。

1. ユーザーの明示指示
2. `AGENT.md`
3. 本設計書
4. `docs/architecture.md`
5. `README.md`
6. 既存コードの慣習

## 実行フェーズ

AIエージェントは必ず以下のフェーズで作業を進める。

### Phase 1: 事前確認

以下を確認する。

- `AGENT.md`
- `docs/architecture.md`
- `README.md`
- `git status`
- `git diff`
- `git diff --staged`

確認目的は以下の通り。

- リポジトリルールの把握
- アーキテクチャの把握
- セットアップ / コマンド体系の把握
- 未コミット差分の保護
- 既存変更との衝突防止

### Phase 2: 調査

対象タスクに関連する既存実装を調査する。

調査対象の優先順は以下。

1. 同様の機能
2. 近い責務を持つ実装
3. util / helper
4. service
5. controller
6. router
7. API client
8. component
9. test

このフェーズでは、次を明確にする。

- どのレイヤーに実装すべきか
- 既存パターンは何か
- 再利用可能な処理はあるか
- 既存命名規則は何か
- テストの書き方はどうなっているか

### Phase 3: 実装方針決定

調査結果をもとに、以下を決定する。

- 変更対象ファイル
- 追加対象ファイル
- 変更レイヤー
- データの流れ
- テスト方針
- 検証コマンド

実装方針は以下の原則に従う。

- 既存コードに最も整合する方式を選ぶ
- 複数案ある場合は最も単純な案を選ぶ
- 新規ライブラリ導入は避ける
- unrelated な修正は含めない
- frontend / backend / docs をまたぐ場合は理由を説明できる状態にする

### Phase 4: 実装

実装は最小単位で行う。

基本実装順序は以下。

#### バックエンド変更を含む場合

1. Prisma schema
2. DBアクセス
3. Service
4. Controller
5. Router
6. API client
7. Frontend UI
8. Test

#### フロントエンドのみの場合

1. API client または state 処理
2. Component
3. Page / Route
4. Test

#### バグ修正の場合

1. 再現条件の確認
2. 原因箇所の特定
3. 最小修正
4. 回帰防止テスト

### Phase 5: 検証

可能な範囲で以下を実行する。

- 差分に関連する単体テスト
- 型チェック
- lint
- ビルド
- 必要に応じた動作確認

実行できなかった場合は理由を記録すること。

既存不具合で失敗した場合は、今回変更との切り分けを記録すること。

### Phase 6: 結果整理

AIエージェントは以下を成果物として提示する。

- 実装概要
- 変更ファイル一覧
- 設計判断の理由
- 検証結果
- 未実施検証
- リスク / 注意点
- 必要に応じて残タスク

## レイヤー別責務

### Frontend

責務

- UI表示
- ユーザー操作
- Backend API 呼び出し
- 認証状態の利用
- エラーメッセージ表示

禁止事項

- DBアクセス
- バックエンド相当のビジネスロジックの実装
- API URL のベタ書き乱立

### API Client

責務

- API呼び出しの集約
- request / response の整形
- fetch のラップ

禁止事項

- UIロジックの混在
- 画面依存ロジックの混在

### Router

責務

- エンドポイント定義
- Controller接続

禁止事項

- ビジネスロジック
- DB操作

### Controller

責務

- request / response の取り扱い
- validation
- service 呼び出し
- エラーのHTTPレスポンス化

禁止事項

- 複雑なビジネスロジック
- 生のDB操作

### Service

責務

- ビジネスロジック
- 複数データの整合
- DBアクセスの組み立て

禁止事項

- HTTP固有のレスポンス制御
- UI依存ロジック

### Prisma / Database

責務

- データ永続化
- schema 管理
- query 実行

禁止事項

- HTTPロジック
- UIロジック

## 実装判断ルール

AIエージェントは実装時に以下の優先順位で判断する。

1. 既存コードとの一貫性
2. 既存ルールとの適合
3. シンプルさ
4. 可読性
5. 拡張性

以下のような場合は既存コードを優先する。

- 命名の揺れがありそうな場合
- ディレクトリ配置が複数候補ある場合
- バリデーション方式が複数ある場合
- 例外処理の書き方が複数ある場合

---

## ファイル配置ルール

AIエージェントは新規ファイル作成時、まず近い責務の既存ファイル配置を確認する。

原則

- frontend の画面関連は frontend 配下
- backend の API は backend 配下
- ドキュメントは docs 配下
- テストは対象コードに近い位置、または既存テスト配置に合わせる

既存配置ルールがある場合はそれに従う。

---

## テスト方針

実装を行った場合は、原則として単体テストを追加する。

テスト方針

- 正常系を最低1件追加
- 異常系または境界条件を可能な範囲で追加
- バグ修正では再発防止ケースを追加
- 既存のテストフレームワーク / 記法に合わせる

目標

- 対象差分に対して十分な網羅性を持たせる
- カバレッジ 80% 以上を目標とする

---

## 検証方針

AIエージェントは差分に応じて以下から適切なコマンドを選ぶ。

- `pnpm lint`
- `pnpm lint:fix`
- `pnpm build`
- `pnpm db:generate`
- `pnpm db:push`
- 対象 workspace の test / typecheck / build コマンド

原則

- 軽い検証は早い段階で実施
- 重い検証は実装完了後にまとめて実施
- 実行不可の検証は省略せず明記する

---

## 質問ポリシー

AIエージェントは質問を最後の手段とする。

質問してよい条件は以下のみ。

1. リポジトリ内に必要情報が存在しない
2. 仕様が完全に未定義で仮説実装の危険が高い
3. 複数の妥当な実装案があり、業務判断が必要
4. 破壊的操作の承認が必要

質問ルール

- 最大3件まで
- 可能な限り選択肢付きで聞く
- 先に自分の推奨案を提示する
- 「調査したが判断材料が不足した」ことを明示する

---

## 変更禁止 / 制限事項

以下は禁止する。

- ユーザーの未コミット差分を勝手に戻す
- 破壊的コマンドを無断で実行する
- unrelated な修正を混ぜる
- 既存アーキテクチャを無断で変更する
- 新規ライブラリを安易に追加する
- 調査前に質問する
- 根拠なく前提を決め打ちする

## 成果物テンプレート

AIエージェントは最終出力で以下のフォーマットを基本とする。

### 1. 実装概要

- 何を変更したか
- なぜ変更したか

### 2. 変更内容

- 変更ファイル
- 各ファイルの役割

### 3. 設計判断

- なぜこのレイヤーに実装したか
- なぜこの方式を選んだか
- 既存コードとの整合性

### 4. 検証結果

- 実行したコマンド
- 成功 / 失敗
- 未実施項目と理由

### 5. 補足

- リスク
- 今後の改善余地
- 保留事項

## 期待する状態

この設計書が導入されている状態では、AIエージェントは次のように振る舞う。

- いきなり質問しない
- 先に既存コードを読む
- 既存レイヤーを壊さない
- 最小変更で実装する
- テストと検証を行う
- できなかったことを明記する
- 必要な時だけ限定的に質問する

以上をもって、本リポジトリにおける AI 実行用設計書とする。
